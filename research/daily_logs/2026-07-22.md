# Daily log — 2026-07-22

## Session goal

Determine whether the Valve Index Controller's LED can be controlled by
software beyond documented SteamVR/OpenVR behavior. Full methodology in
`../../docs/01_project_overview.md`.

## Timeline

**Phase 1 — background research.** Reviewed prior public work: Valve's
`hid-steam.c` kernel driver (haptic command precedent), the
`nairol/LighthouseRedox` project (HID report-ID conventions for the older
Vive wand), `DJm00n/ControllersInfo` (checked, no Valve entries — dead
end), Valve's own `IndexHardware` repo (checked, CAD files only — dead
end). Established initial hardware hypothesis (nRF52840 MCU, iCE40 FPGA,
Winbond flash) from public teardown sources. Full account:
`../../docs/02_background.md`.

**Phase 2 — firmware acquisition.** Located firmware files shipped
directly inside the SteamVR installation
(`drivers/indexcontroller/resources/firmware/indexcontroller/`). Attempted
plain `zlib.decompress()` against each file — **succeeded immediately, no
protection to bypass.** This eliminated the anticipated hardware-extraction
contingency (nRF52840 APPROTECT bypass research, done as background prep
but never needed). Full account: `../../docs/04_firmware_acquisition.md`,
`../../docs/13_experiments.md` Experiment 4.

**Phase 3 — static string analysis.** Searched decompressed firmware
strings for LED-related content (fixed an early false-positive bug: naive
`grep -i "led"` matched "Failed"/"failed" everywhere; switched to
word-boundary matching). Found `led.c`, `led_driver_lp5562.c`,
`led_driver_gpio.c`, `led_policy_knuckles.c` source filenames, and the key
log format string `"LED %u: color = 0x%06X->0x%06X, status: %u, blink_on =
%s"`, confirming a TI LP5562 RGBW driver chip and a generic
arbitrary-color-capable firmware function. Full account:
`../../docs/08_lp5562_driver.md`.

**Phase 4 — live USB HID investigation.** Systematically enumerated every
HID report on every controller interface
(`scripts/dump_hid_descriptors.py`, `scripts/parse_hid_descriptor.py`,
`scripts/read_feature_reports.py`). Tested `SET_FEATURE` on single-byte
Feature-report candidates
(`scripts/test_feature_toggle_candidates.py`) — **report `0x12` set to
`0x01` triggered device reset and exposed a previously undocumented
4th "Debug" HID interface.** This is the single most significant discovery
of this early phase. Full account: `../../docs/12_debug_interfaces.md`,
`../../docs/13_experiments.md` Experiments 1–2.

**Phase 5 — debug shell protocol reverse engineering.** First attempt
(`scripts/probe_debug_shell.py`) used incorrect wire framing (text starting
immediately after the report-ID byte) and produced a truncated response
(`"Unknown Command elp"` for `"help"`). Diagnosed the framing bug from the
response's own length byte, fixed it, and retrieved the full command list.
`scripts/debug_shell.py` is the resulting reusable client. Full account:
`../../docs/12_debug_interfaces.md`, `../../docs/13_experiments.md`
Experiment 3.

**Phase 6 — live config exploration.** Surveyed every debug shell command
(`../../docs/11_hid_commands.md`). Confirmed the `config` command exposes
46 live values including `led_driver_current_{r,g,b,w}` (default `8`
each). Confirmed `config` is read-only via this shell — several write
syntaxes were tried and all silently no-op'd (dead end, documented in
`../../docs/14_failed_attempts.md`).

## Where the session left off

Progress was substantial: firmware extracted trivially, LED driver chip
identified with high confidence, a major undocumented HID interface
discovered and its protocol reverse engineered, and a confirmed (if not
yet solved) path toward writable config values. The session paused with
the write-path question open — resumed the next day, see
`2026-07-23.md`.
