## Addendum — remote follow-up session (repo made public, Patch C designed)

Continuation of the same investigation, from a session with no physical
access to the test controller (the human tester was away from the hardware).
No live hardware experiments were possible; work was limited to repository
maintenance and file-level (unflashed) patch design.

**Repository made public.** The archive created at the end of the prior
session was pushed to a public GitHub repository. This surfaced a privacy
question: the real controller serial number (used throughout the docs as a
concrete example value) was now publicly exposed real-world hardware
identifying information with no research value beyond being *a* valid
example. Redacted to a placeholder (`LHR-XXXXXXXX`) across all 9 files that
referenced it.

**Dead-end visibility strengthened.** At the human tester's explicit
request ("document stuff we have done, but it failed, so others know it's
a dead end"), added a dedicated "What didn't work" section to the top-level
[`README.md`](../../README.md), surfacing headline dead ends inline rather than relying on a
reader to find [`../../docs/14_failed_attempts.md`](../../docs/14_failed_attempts.md) via the repository-layout
listing alone.

**Patch C designed: forced solid blue.** The human tester asked whether the
LED could be forced to a color the controller doesn't normally show (e.g.
purple), while away from hardware. Analysis of the proven black patch's
exact mechanism ([`docs/15_firmware_patching.md`](../../docs/15_firmware_patching.md) §15.3) showed that the same
2-byte-for-2-byte instruction swap, with a nonzero immediate instead of
zero, can reach any pure-blue shade (0-255 intensity) — because the
underlying Thumb instruction (`movs Rd, #imm8`) zero-extends an 8-bit value
into the whole 32-bit register, and this firmware packs Blue in the
lowest byte. Built [`scripts/patch_led_solid_color.py`](../../scripts/patch_led_solid_color.py), generated and
file-verified (round-trip decompress/patch/recompress/footer-rebuild) a
`.fw` output at maximum blue intensity — **but could not flash or visually
verify it, since no hardware was available this session.** Documented as
explicitly UNTESTED in three places ([`docs/15_firmware_patching.md`](../../docs/15_firmware_patching.md),
[`docs/18_future_work.md`](../../docs/18_future_work.md) new Priority 1, [`hashes/firmware_hashes.txt`](../../hashes/firmware_hashes.txt)) so
this distinction isn't lost.

**True purple confirmed harder, not attempted.** The same analysis showed
purple (needing Red *and* Blue simultaneously) cannot be reached by the
same single-instruction technique — it needs a full 32-bit constant load,
which doesn't fit the original 2-byte instruction slot without relocating
surrounding code (a "code cave"). Designed only at the concept level;
written up as [`docs/18_future_work.md`](../../docs/18_future_work.md) new Priority 2 with the specific
open questions (available safe space in the image, branch-encoding range)
rather than attempted, since it's a meaningfully bigger undertaking than
anything already proven and shouldn't be rushed without hardware access to
test against.

**Next session, first task:** flash `indexcontroller_app_LEDBLUE_255.fw`
and observe the result — see [`docs/18_future_work.md`](../../docs/18_future_work.md) Priority 1.

## Addendum 2 — resumed charging-connection research, controller still connected

Continuation with the physical controller still connected to the host
(providing HID/debug-shell access) but still no visual observation
possible. The human tester asked explicitly to push further on the
selective (charging-aware) patch and to get as close to "test ready" as
tools alone would allow.

Confirmed hardware still healthy and in the same state as the previous
addendum (debug interface still active, stock firmware, no reset since
last flash).

Picked up the data-flow approach recommended in
[`docs/16_charging_led_research.md`](../../docs/16_charging_led_research.md). Found the power-management state
struct's exact RAM base address (`0x2000378c`, confirmed via all four
known PM functions' literal pool operands), then found 14 additional raw
references to that same address elsewhere in the firmware — a technique
independent of, and more robust than, searching for a *function's* address
(since it doesn't depend on the referencing code itself being
call-graph-reachable). Investigating those 14 references found new struct
field offsets (`+0x15`, `+0x16`, `+0x24`, `+0x28`), and one lead that
looked very promising (`0x414dbc`/`0x414e38`, called both by PM code and
by the LED subsystem) but resolved to a dead end once decompiled — a
generic `tick_handler.c` timer facility with 19–20 unrelated callers, not
an LED-specific mechanism.

The three newly-found PM-adjacent functions (`+0x28` checker, `+0x15`
writer, `+0x16` writer) show the same "zero statically-findable direct
callers" symptom already documented for the LED Layer-3 functions. This
generalizes the earlier finding: the obstacle is very likely one shared,
firmware-wide dispatch mechanism affecting multiple unrelated subsystems,
not something LED-specific — a real, if incomplete, architectural insight.

**A genuine tooling bug was found and fixed along the way.** While
verifying one of the above leads, a "caller" found by the project's
brute-force `BL`-instruction-encoding search turned out, on manual
cross-check with `capstone`, to actually call a different address — the
search had a bit-shift bug causing occasional false-positive aliasing.
Fixed, committed as [`scripts/find_bl_callers.py`](../../scripts/find_bl_callers.py) (this technique
previously existed only as an uncommitted inline script), and re-run
against the project's most important prior "zero callers" findings (the
LED Layer-3 functions) to confirm they still hold under the corrected
tool — they do. This is exactly the kind of self-correction the project's
own methodology calls for: caught, fixed, documented, and the dependent
conclusions explicitly re-verified rather than assumed safe.

**Honest status: still not "test ready" for a selective patch.** No code
was found that reads the PM state enum, so there is still nothing to
conditionally branch a patch on. This session added real, concrete facts
(struct address, more field offsets, one ruled-out lead, one corrected
tool, and a sharper hypothesis about a shared dispatch mechanism) but did
not produce a new patch. The only patch that is "test ready" right now
remains Patch C (forced blue) from Addendum 1 above — see
[`docs/18_future_work.md`](../../docs/18_future_work.md) Priority 1.

Full technical detail: [`docs/16_charging_led_research.md`](../../docs/16_charging_led_research.md) "Follow-up
session", [`docs/06_firmware_symbols.md`](../../docs/06_firmware_symbols.md) §6.5, [`docs/14_failed_attempts.md`](../../docs/14_failed_attempts.md)

## Addendum 3 — corrected a wrong assumption, opened a third research thread

The user corrected a claim this project had gotten wrong: blue is not an
unused color, it's a real, existing state used for USB/host-connection
status and pairing-mode indication (blinking). The user also confirmed the
complete known color palette (green, red, blue, white, orange) and
proposed a new, potentially more tractable target for the selective patch:
disable the LED specifically while connected to SteamVR, revert to stock

Corrected throughout the repository — see the new "Corrected assumptions"

table in [`docs/09_led_policy.md`](../../docs/09_led_policy.md). The correction does not affect Patch C's
mechanism, only the claim about why the color it produces was interesting.

Investigated the "connected to host" angle as a third research thread this
session (after the charging-enum trace and the timer-facility dead end
earlier today). Traced the live `battery` command's USB-status strings
(hit the same recurring zero-cross-references symptom already documented),
chased a "pairing" string match that turned out to be a red herring (a
config key name, `pairing_button_press_ms`, not pairing-state logic), but
that dead end led somewhere useful: the boot-time mode-announcement
function it belonged to has a struct base address only 20 bytes from the
already-known PM struct. Following references to that address found a
single, very large initialization function touching offsets beyond
`+0x1000` — strong evidence that the PM/charging struct and this
pairing/mode struct are not separate at all, but part of one large,
shared, multi-kilobyte device-state block.

**This is a genuinely new, promising lead, not a solved problem.** Mapping
a structure this large is a bigger undertaking than anything completed so
far in this project and was not finished this session. Stopped here
deliberately rather than continuing to spawn open-ended investigation —
three separate threads (charging enum, timer facility, connection state)
have now each surfaced real findings without closing the central question,
and the honest status remains: **no selective patch yet, Patch C (forced
blue) is still the only test-ready artifact.**

## Addendum 5 — a multi-tool sweep found the PM state enum's reader, and a concrete dispatch hypothesis

Asked directly what other RE tools might help beyond Ghidra, and then to
actually try the relevant ones (hardware SWD debugging — the single
highest-leverage option — was named but explicitly ruled out as
unavailable right now). Installed and ran **angr**, **Unicorn Engine**,
and revisited **radare2** (previously tried on this exact firmware early
in the project and largely abandoned, [`tools/radare2_setup.md`](../../tools/radare2_setup.md)) with a
narrower technique, plus a small DIY version-diff across this project's
own locally-held firmware builds. IDA Free and Binary Ninja were
deliberately not attempted — both require a login-gated vendor download,
not appropriate to do autonomously.

**Real progress, not just more negative confirmation:** a systematic
angr-based literal-pool scan (checking the PM struct's full offset range,
where the original Ghidra-era search had only checked the base pointer)
found 5 new reference sites the earlier investigation missed. One led
directly to `sub_422f21` (`0x422f21`) — the first code this project has
located that actually *reads* the state enum at `+0xc`, previously
documented as having no known reader at all
([`docs/06_firmware_symbols.md`](../../docs/06_firmware_symbols.md) §6.5). It reads the enum, and for
input values `{0,1,2,5}` writes it to `3` (an idempotent "already
transitioned" pattern) before calling into two functions that build small
opcode-tagged buffers (`0x40`, `0x2`, `0x53` observed) and hand them to
apparent common dispatcher functions (`0x41f56d`, `0x4153e5`) — not a
direct call to anything LED-related. **Verified empirically, not just by
eye:** a Unicorn harness with the function's own `bl` calls NOP-patched
out (so only its branching logic runs) confirmed this exact behavior
across a swept range of initial state values.

This produces a real hypothesis for *why* four independent
techniques now (Ghidra, this project's own `BL`-search, angr, and this
session's radare2 retry) all agree on zero direct callers for the LED
Layer-3 functions: if LED updates are driven by this same kind of
message/event dispatch rather than a direct function call, there simply
is no direct call for any of these techniques to find — the absence isn't
a tooling gap, it's consistent with the actual mechanism. **This is not
confirmed** — it hasn't been shown that the LED subsystem is on the
receiving end of this specific mechanism, only that the PM code
demonstrably uses it for at least some of its own side effects. Decoding
what `0x41f56d`/`0x4153e5` actually do with their opcode argument is now
the top recommendation for this thread, ahead of the "map the whole
shared struct" idea from the previous session.

A version-diff sanity check (DIY, Diaphora/BinDiff not installed — judged
not worth the setup risk for one targeted comparison) confirmed
`sub_422f21` and the PM struct's exact layout are present, byte-identical,
in the `ev` variant firmware (2023-10-13) — a genuinely separate build
from the primary target — and that a structurally similar opcode pattern
exists in both 2019-era builds too, at their own (different) addresses.

**Honest status: still open.** The core question — what connects PM state
to LED color — is not solved. What changed is real: the project now knows
*specifically* what reads the state enum, has empirical (not just static)
confirmation of its behavior, and has a much more concrete, addressable
hypothesis (decode two specific dispatcher functions) in place of a vague
"probably a shared-state RTOS thing." Also caught and fixed, mid-session,
a small file-offset bug of its own (forgot to strip the Thumb-mode address
tag before computing a raw file offset, producing garbage disassembly
until noticed and corrected — documented inline in
[`research/decompiler_notes/13_unicorn_state_handler_sweep.py`](../decompiler_notes/13_unicorn_state_handler_sweep.py)).
