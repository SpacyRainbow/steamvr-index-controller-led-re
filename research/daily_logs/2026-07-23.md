successfully.

**LED-current value patches flashed live.** value=255: flashed
successfully, but the human tester reported no clearly perceptible
brightness change (inconclusive). value=0: flashed successfully, produced
dramatic dimming but not full black, and the boot self-test's color-cycling
behavior was still visible. Full account: `../../docs/13_experiments.md`
Experiment 6b–6c.

**Operational incident:** a recursive `inotifywait -m -r /tmp` watcher
(started to look for temp files created by the update tool) fed back on
its own output file, filling the sandbox's tmpfs to 31 GB before being
noticed and fixed. No impact on the controller or any research findings;
documented as a pure tooling caution in
`../../docs/14_failed_attempts.md`.

**Layer-1 code patch — the project's central result.** Static analysis
located the exact instruction (`0x41dc20`, inside the low-level LP5562
PWM-write function) where the final computed color is captured before
being split into hardware register writes. Patched it
(`scripts/patch_led_black.py`) to force this to always be zero. Flashed
live. **Human tester confirmed: "the LED is off!!"** This is the project's
primary positive, unambiguous result. Full account:
`../../docs/13_experiments.md` Experiment 7,
`../../docs/15_firmware_patching.md`.

**Follow-up request: selective (charging-aware) patch.** The human tester
asked for a refinement — keep charging/charged LED indication, blank the
LED only during normal use. This proved substantially harder. Traced
three layers of the LED call graph, correcting an earlier function-
boundary misidentification along the way. Installed a full Ghidra headless
toolchain (without root — `../../tools/ghidra_setup.md`) after radare2's
automatic analysis proved insufficient for the ARM firmware's dense
VFP-heavy code regions. Ghidra immediately resolved several previously
unresolvable questions (correct wrapper function boundary, real callers of
the low-level writer), but the trace ultimately stopped at a genuine open
question: the power-management task's charging-state code does not
directly call into the traced LED call graph, implying an RTOS
shared-state connection not yet located. Full account:
`../../docs/16_charging_led_research.md`, with every specific dead end
along the way preserved in `../../docs/14_failed_attempts.md`.

**Session conclusion.** The core research question is answered with high
confidence (yes, software can control the LED — proven live). The
selective-patch refinement remains open. At the human tester's request,
this repository was then created to preserve the full investigation for
future continuation.

## Addendum — remote follow-up session (repo made public, Patch C designed)

Continuation of the same investigation, from a session with no physical
access to the test controller (the human tester was away from the hardware).
No live hardware experiments were possible; work was limited to repository
maintenance and file-level (unflashed) patch design.

**Repository made public.** The archive created at the end of the prior
session was pushed to a public GitHub repository. This surfaced a privacy
question: the real controller serial number (used throughout the docs as a
concrete example value) was now publicly exposed real-world hardware
identifying information with no research value beyond being *a* valid
example. Redacted to a placeholder (`LHR-XXXXXXXX`) across all 9 files that
referenced it.

**Dead-end visibility strengthened.** At the human tester's explicit
request ("document stuff we have done, but it failed, so others know it's
a dead end"), added a dedicated "What didn't work" section to the top-level
`README.md`, surfacing headline dead ends inline rather than relying on a
reader to find `../../docs/14_failed_attempts.md` via the repository-layout
listing alone.

**Patch C designed: forced solid blue.** The human tester asked whether the
LED could be forced to a color the controller doesn't normally show (e.g.
purple), while away from hardware. Analysis of the proven black patch's
exact mechanism (`docs/15_firmware_patching.md` §15.3) showed that the same
2-byte-for-2-byte instruction swap, with a nonzero immediate instead of
zero, can reach any pure-blue shade (0-255 intensity) — because the
underlying Thumb instruction (`movs Rd, #imm8`) zero-extends an 8-bit value
into the whole 32-bit register, and this firmware packs Blue in the
lowest byte. Built `scripts/patch_led_solid_color.py`, generated and
file-verified (round-trip decompress/patch/recompress/footer-rebuild) a
`.fw` output at maximum blue intensity — **but could not flash or visually
verify it, since no hardware was available this session.** Documented as
explicitly UNTESTED in three places (`docs/15_firmware_patching.md`,
`docs/18_future_work.md` new Priority 1, `hashes/firmware_hashes.txt`) so
this distinction isn't lost.

**True purple confirmed harder, not attempted.** The same analysis showed
purple (needing Red *and* Blue simultaneously) cannot be reached by the
same single-instruction technique — it needs a full 32-bit constant load,
which doesn't fit the original 2-byte instruction slot without relocating
surrounding code (a "code cave"). Designed only at the concept level;
written up as `docs/18_future_work.md` new Priority 2 with the specific
open questions (available safe space in the image, branch-encoding range)
rather than attempted, since it's a meaningfully bigger undertaking than
anything already proven and shouldn't be rushed without hardware access to
test against.
